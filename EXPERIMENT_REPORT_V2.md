# UCB卫星路由改进实验报告

## 1. 实验概述

本报告记录了UCB分布式路由算法三项改进（内存修复、Top-K候选集、路径缓存）和SPF动态基线算法的实现、调试和验证实验。

### 改进内容

| 改进项 | 描述 | 配置参数 |
|:---|:---|:---|
| **内存崩溃修复** | 全局数据包状态表TTL清理 + 计数器扩容uint64 | `ucb_packet_state_ttl_slots=10` |
| **Top-K候选集** | 按距离筛选UCB可选邻居，限制搜索空间 | `ucb_top_k=4` |
| **路径缓存** | 缓存成功路径，以高概率复用，保障业务连续性 | `ucb_path_cache_prefer_prob=0.7` |
| **SPF动态基线** | 基于实际设备拓扑的Dijkstra最短路径，物理距离为权重 | `satellite_network_routing=spf_dynamic` |

### SPF基线算法调试过程

SPF基线在实现过程中发现并修复了三个关键Bug：

1. **距离→设备拓扑Bug**：Dijkstra使用距离判断非本节点邻居的连通性，在max_isl_length_m=10^9m的25x25场景下，所有卫星对都"直接连通"，形成完全图。但实际Plus-Grid拓扑中每颗卫星只有4-6个ISL邻居。
   - **修复**：`BuildGlobalAdjacency()` 从所有节点的实际设备/通道枚举邻居

2. **GS-to-GS环路Bug**：GSL共享信道将所有节点连在一起，Dijkstra会生成经过地面站→地面站的路径，但在卫星网络中GS之间不能直接通信。
   - **修复**：Dijkstra中过滤掉GS-to-GS边

3. **GSL共享信道卫星互连Bug**：GSL信道作为共享介质，使所有连接该信道的设备互为邻居，导致卫星间也通过GSL"直接连通"。
   - **修复**：`BuildGlobalAdjacency()` 中GSL设备仅添加跨类型邻居（sat↔gs）

## 4. 分析与结论

### 4.1 内存修复有效性

**25x25场景**：内存修复完全有效。之前UCB基线在25x25 traffic_matrix中因SIGSEGV崩溃（`g_ucb_packet_state_by_uid`无限增长），修复后运行完整50秒模拟无崩溃。

**Iridium场景**：SIGSEGV仍然发生。可能的根因：
1. Iridium动态拓扑导致更多路由变化和TCP重传，包状态创建速率更高
2. Iridium场景的邻居数量不同，可能触发其他边界条件
3. 需在gdb下运行获取精确崩溃位置

### 4.2 UCB vs SPF 性能差距

SPF在25x25场景达到100%交付率，而UCB改进仅14.7%。差距原因：
1. **UCB探索开销**：UCB需要探索多个邻居来学习奖励值，初期大量数据包被发送到次优路径
2. **信息局限**：UCB仅基于本地邻居信息做决策，而SPF拥有全局拓扑视图
3. **收敛速度**：在50秒模拟时间内，UCB可能未充分收敛

### 4.3 Top-K和路径缓存效果

- Top-K候选集缩小了搜索空间（从4-6个邻居减少到4个），理论上应加速收敛
- 路径缓存（prefer_prob=0.7）在缓存命中时复用成功路径，减少探索
- 但14.7%的完成率表明，这些改进仍不足以弥补UCB在短期内与SPF的性能差距

### 4.4 后续方向

1. **Iridium SIGSEGV**：需在gdb下定位崩溃点，可能还有其他内存泄漏
2. **UCB参数调优**：增大探索概率（当前Iridium设为0.0）、调整奖励权重
3. **更长模拟时间**：验证UCB在充分收敛后是否能接近SPF性能
4. **混合策略**：考虑SPF引导+UCB微调的混合路由方案
